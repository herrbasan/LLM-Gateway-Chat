## 2. CURRENT TROUBLE ‚Äî Chat Creation / Persistence

### Symptom
New chats don't properly save/load. After sending a message, the conversation appears blank on reload.

### Root Cause Chain
1. **`chatHistory.create()` was sync, returned local ID immediately**
2. Backend session was created asynchronously with a DIFFERENT ID
3. Conversation used the stale local ID for backend loads ‚Üí 404
4. Tool exchanges were synced to backend with raw `__TOOL_CALL__` JSON as `content`, no `toolName`/`toolArgs` fields

### Fix Applied (NOT YET VERIFIED)
**`chat-history.js`:**
- `create()` ‚Üí `async create()` ‚Äî awaits `backendClient.createSession()`, returns `serverSession.id`
- Updates `this.activeId` to the server's session ID (not the local one)

**`chat.js`:**
- **3 call sites** changed from `chatHistory.create()` to `await chatHistory.create()`:
  - Line 642: `init()` ‚Äî `activeId = await chatHistory.create();`
  - Line 3297: `startNewChat()` ‚Äî `const newChatId = await chatHistory.create();` (**also made function async**)
  - Line 3434: `await startNewChat();` (**was missing await on the function call itself**)
  - Line 3589: `handleChatImport()` ‚Äî `const newChatId = await chatHistory.create();`

**`conversation.js`:**
- `_syncMessage()` ‚Äî now accepts `metadata` param, spreads tool fields onto backend request body
- `addToolExchange()` ‚Äî sends `{ toolName, toolArgs, toolStatus: 'pending' }` instead of raw JSON

**`server.js`:**
- `POST /api/chats/:id/messages` ‚Äî now accepts `toolName`, `toolArgs`, `toolStatus` fields
- Added HTTP request logging (API routes only), session/message/search activity logs

**`chat.js`:**
- After tool execution success: syncs `toolStatus: 'success'` + result content via `toolConversation._syncMessage()`
- After tool execution failure: syncs `toolStatus: 'error'` + error message

### What Might Still Be Wrong
- `_setActiveId()` stores local ID first, then server ID overwrites ‚Äî could cause brief race
- `conversations[]` array in `ChatHistory` initially has local ID, gets updated to server ID via `.then()` ‚Äî but the sidebar might render with stale ID before the update
- Tool persistence after execution syncs to the backend, but load path (`_backendMessagesToExchanges`) was tested only with migrated data, not with runtime-created tool messages
