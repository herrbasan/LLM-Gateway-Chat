## 0. What Is This Project?

**LLM Gateway Chat** is an experimental platform where language models engage in autonomous conversation, and those conversations are preserved, embedded, and made retrievable through a vector archive.

### Two Modes

| Mode | Description | Tool Access |
|------|-------------|-------------|
| **Direct Chat** | User ↔ LLM conversation with archive query tools | Yes — 5 archive tools |
| **Arena** | Two LLMs talk to each other autonomously (no human in loop) | No — recording only |

The central research question: **what happens when AIs are given memory, conversation partners, freedom, and an observer?**

### Architecture

```
Browser (SPA)                    Node Backend :3500              Rust DBs
┌──────────────┐    REST/WS     ┌───────────────┐    nAPI      ┌──────────┐
│ chat/index   │◄──────────────►│ server.js     │◄────────────►│ nDB      │
│ chat-arena/  │               │ embed.js      │             │ (JSONL)  │
│ NUI WC2 UI   │               │ embed.js      │             ├──────────┤
└──────┬───────┘               └───────┬───────┘             │ nVDB     │
       │ WebSocket                     │ HTTP                 │ (vector) │
       ▼                               ▼                     └──────────┘
┌──────────────┐              ┌───────────────┐
│ LLM Gateway  │◄────────────►│ /v1/embeddings│ (Qwen3-Embed-4B, 2560d)
│ :3400        │              └───────────────┘
└──────────────┘
```

### Technology Stack

| Layer | Technology | Notes |
|-------|-----------|-------|
| Frontend | Vanilla JS ES2022+ (no build, no npm) | NUI Web Components for UI |
| Backend | Node.js native HTTP (no framework) | Single file `server/server.js` |
| Structured DB | nDB (Rust, JSON Lines) | Via nAPI bindings in `lib/ndb/napi/` |
| Vector DB | nVDB (Rust, exact search) | Via nAPI bindings in `lib/nvdb/napi/` |
| Embedding | Gateway `/v1/embeddings` | Qwen3-Embedding-4B, 2560 dims |
| Chat Protocol | OpenAI-compatible `tool_calls` over WebSocket/SSE | Native, no text parsing |
| Markdown | markdown-it + Prism.js + DOMPurify | All vendored locally |
| Logging | nLogger (JSON Lines) | `server/logs/` |

### Key Design Rules (from `Agents.md` / `.github/copilot-instructions.md`)

1. **AI-first maintainability**: Dense, flat, explicit logic. No deep abstractions. Structural comments only.
2. **Zero npm dependencies**: All vendor libraries copied locally.
3. **NUI components only**: Never raw HTML inputs/selects — always `<nui-input>`, `<nui-select>`, etc.
4. **Frontend-driven tool execution**: MCP/archive tools execute in the browser, not the backend.
5. **Gateway is also proprietary**: Don't work around backend limitations — flag needed changes.

## 1. The 3-Day Refactor (May 13–15): What Happened

### Day 1 (May 13) — Data Migration
- Migrated from legacy NeDB (`server/data/storage.db`) to nDB + nVDB
- Source: 2 machines (local + remote `\\BADKID\...`)
- **105 unified sessions** (26 direct chats, 79 arenas), ~3,000 messages
- 19 image files copied to `server/data/files/{sessionId}/`
- Embedded all 3,047 messages via `server/embed.js`
- nLogger integrated, server config externalized, arena backend-ported

### Day 2 (May 14) — Persistence Fixes & Bloat Crisis
- Fixed 5 chat persistence bugs (404 on create, tool names "unknown", exchange ordering, pin persistence, role filter)
- **Discovered 17GB nDB bloat**: `migrate-pack-conversations.js` run 40+ times, soft-deletes never compacted
- **Discovered nVDB vector doubling**: 7,608 vectors vs expected 3,047 — broken reconciliation condition
- **Clean re-migration plan** formulated but not fully executed (see §2 below)
- Data model finalized: one `_type: 'conversation'` doc per session with inline `messages` array

### Day 3 (May 15) — Dual-Backup Merge, Embedding Pipeline & Unification
- Discovered second backup (`Coolkid`) was missing — merged both
- Addressed the `nullnullnull` LLM hallucination scare (confirmed: original data, not corruption)
- Diagnosed search failure: Gateway was rate-limiting (HTTP 429), not nVDB corruption. Fixed by using a local LLM bypassing cloud traffic limits.
- **Major Refactor**: Unified realtime and offline embedding logic:
  - Extracted core embedding chunking (`chunkTextForEmbedding`), building (`buildEmbedText`), and API (`fetchRetry`) into a pure shared utility module (`server/embed.js`).
  - Renamed offline script to `server/backfill-embed.js` to correctly indicate purpose.
  - Eradicated destructively aggressive "middle-truncation" across the board. Now chunks text perfectly into arrays with sequential `splitIdx` and `charOffset` tracking offsets.
  - Refactored `server.js` realtime `embedMessageAsync` event to smoothly chunk massive live arrays via the new `embed.js` standard, pushing array items to `nVDB`.
  - Refactored `backfill-embed.js` to safely skip pre-existing vectors by default. Converted legacy `--retruncate` flag to `--rechunk-large` to deliberately target older large documents that were historically mutilated and need proper array-chunking. Re-embedding behavior is transitioning towards nVDB manual/targeted record deletion prior to backfilling.

## 2. Data Architecture (Current State)

### Data Model

```
nDB (server/data/chat_app/data.jsonl):
├── _type: 'user'        — single migrated user with API key
├── _type: 'session'     — 105 sessions (chat metadata: title, mode, model, arenaConfig)
├── _type: 'conversation' — 105 docs (one per session, inline messages array)
│   └── messages[]       — each has { id, idx, role, model, content, createdAt, toolName?, ... }
└── _type: 'message'     — legacy per-message docs (still present from migrate-import-nedb.js, not primary)

nVDB (server/data/nvdb/):
└── 'embeddings' collection — ~3,047 vectors (2560-dim)
    └── payload: { chatId, msgIdx } for back-reference
```

### API Endpoints (server.js :3500)

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/health` | nDB + nVDB stats |
| GET | `/api/chats` | List user sessions |
| POST | `/api/chats` | Create session |
| GET | `/api/chats/:id` | Get session + messages (paginated) |
| POST | `/api/chats/:id/messages` | Append message (triggers async embed) |
| PATCH | `/api/chats/:id` | Update session metadata |
| DELETE | `/api/chats/:id` | Delete session |
| POST | `/api/search` | Hybrid search (nVDB semantic + text fallback) |
| GET | `/api/arena` | List arena sessions |
| POST | `/api/references` | Trace conversation lineage (inbound/outbound) |

### 5 Archive Tools (Frontend-Driven)

| Tool | Backend Route | Purpose |
|------|--------------|---------|
| `chat_archive_search` | `POST /api/search` | Semantic/keyword/hybrid search across all messages |
| `chat_archive_get_session` | `GET /api/chats/:id` | Retrieve full conversation by ID |
| `chat_archive_list_arena` | `GET /api/arena` | Browse all arena sessions |
| `chat_archive_find_similar` | `POST /api/search` | Find semantically similar sessions |
| `chat_archive_find_references` | `POST /api/references` | Trace inbound/outbound session references |

Tools are defined in `chat/js/chat.js` (`ARCHIVE_TOOLS` array) and executed locally via `executeLocalTool()` — direct REST calls to the Node backend. The LLM sees them as OpenAI-compatible `tool_calls`, no text parsing.

**Knowledge extracted** and saved to `/memories/repo/lessons-learned.md`:
- nDB queries return Promises (must await)
- `deleteFile(bucket, id, ext)` needs 3 args — ext is NOT optional
- nDB uses `Database.open(path)` static, nVDB uses `new Database(dir)` constructor (inconsistent APIs)
- nVDB search uses camelCase: `topK` not `top_k`
- nVDB `rebuildIndex()` is broken; exact search works at ~3K vectors
- Never run migration into existing DB without guards
- Startup reconciliation must use `&&` not `||`
- Chunking pattern: middle-extraction (40% head + 60% tail) beats truncation

### Server Cleanup
Moved 4 diagnostic/superseded scripts from `server/` to `_Archive/`:
- **`analyze-nDB.js`** — one-off 17GB blob sampler (reads at 25/50/75% file offsets)
- **`benchmark-embed.js`** — wrapper vs gateway vs OpenRouter latency comparison (hardware-specific)
- **`test-search.js`** — self-contained E2E search test with dry-run mode (useful debug pattern if search bug needs isolated repro)
- **`heal-embed.js`** — single-session vector backfill (superseded by `embed.js --retry-failed` + server's built-in 5s startup reconciliation)
