## 🔴 Critical Issues Found

### 1. nDB File Bloat — 17GB / 40K lines

**Symptom**: `server/data/chat_app` is 17GB with 40,100 lines. nDB.open() takes 50s.

**Root Cause**: `migrate-v2.js` was run multiple times into the same database during debugging. Each run:
1. Deletes existing `_type: 'conversation'` documents → adds ~3MB tombstone lines
2. Creates new conversation documents → adds new ~3MB lines
3. Deletes `_type: 'message'` documents → adds small tombstones

Result: ~40 complete migration cycles × 141 convos × ~3MB average = ~17GB.

nDB's `delete()` is a soft delete — tombstones accumulate in the JSON Lines file forever. `compact()` was never called.

**Active data is intact**: 283 live documents (~20MB). 12,341 soft-deleted tombstones (+ 40K blank/deleted lines).

### 2. nVDB Vector Count Doubled — 7,608 vectors

**Symptom**: nVDB collection has 7,608 vectors vs. expected 3,047. nVDB init takes 183s.

**Root Cause**: Startup reconciliation had a condition bug (`needsEmbed || isStale` instead of `&&`) that re-embedded ALL messages older than 5 minutes on every restart. Also, repeated `embed.js` runs without dropping the collection created duplicates.

Duplicate inserts used the same `msg.id` key — nVDB either updated (no duplication) or inserted duplicates. Need to verify nVDB's insert behavior (upsert vs insert).

### 3. Old Arena Sessions Have No Backend Messages

**Root Cause**: Arena migration created sessions in nDB but messages were only in localStorage/IndexedDB. The store-only backend rewrite deleted the fallback — arena sessions load as empty.

**Data not lost**: Arena messages still exist in the NeDB backups at `D:\Work\CHAT_BACKUP`.

## 🔴 What We Suspect Went Wrong (Root Cause Summary)

1. **Repeated migration runs** — `migrate-v2.js` was executed into the same nDB file multiple times during debugging. Each run deletes and re-creates conversation docs, piling up tombstones.

2. **No compaction** — `db.compact()` was never called. nDB soft-deletes accumulate forever.

3. **Repeated embed runs** — `embed.js` and `heal-embed.js` were run multiple times into the same nVDB collection. Combined with the broken reconciliation condition (`||` instead of `&&`), vectors duplicated.

4. **No pre-condition checks** — Migration scripts don't check if the operation was already performed. They blindly re-execute.
